###### 코딩 테스트 추천 사이트 - 프로그래머스
1. 타인의 풀이를 볼 수 있다.
2. 테스트 케이스를 추가할 수 있다.
###### 타인의 풀이를 보면 사고를 넓힐 수 있다
- 다른 사람이 작성한 코드를 보면 자연스럽게 다양한 문제 풀이 접근 방식이나 코딩 스킬(알고리즘, 입출력, 예외)을 습득할 수 있다.
- 단, 짧게 작성한 숏코딩(Short coding) 이 무조건 좋은 건 아니라는 것을 명심해야한다.
###### 나만의 테스트 케이스를 추가하는 건 좋은 알고리즘을 생각할 때 도움이 된다
- 대부분의 테스트 케이스는 문제를 설명하는 수준에서 제공한다.
- 문제 분석 시 고려해야 할 중요 케이스나 실제 구현 시 실수하기 쉬운 사항을 항상 가정하진 않는다.
- 충분한 시간을 들여 문제를 문석한 다음 코드로 구현하기 전에 여러 예외 상황을 충분히 확인할 수 있도록 나만의 테스트 케이스를 추가해보기를 바란다.
- 특히, 코드를 작성하기 전 문제를 분석하는 단계에서 충분히 예외 테스트 케이스를 추가해볼 것을 권한다.

##### 아는 것과 모르는 것을 명확하게
###### 기록하라
- 모든 문제는 완벽하게 풀 수 없지만 중간까지 갈 수 있다. 여기서 어디까지 생각해보았는지 우선 기록해둔다.
- 기록 할 것: 적용 알고리즘, 근거, 어떻게 코드로 변환할 것인지
- 답안과 기록과 비교하며 복기

###### 시험 보듯 공부하라
- 주기적으로 자체 시험 보기 : 시간 분배 및 긴장감 유도

###### 짧은 시간 공부해서는 절대 코딩 테스트를 통과할 수 없다
- 최소 한 달에서 두 달 정도 매우 집중해서 공부해야 한다

###### 나만의 언어로 요약하라
- 정말 이해했는지 알기 위해서 남이 이해한 내용을 나만의 언어로 요약해보아야 한다. > 글, 그림, 수식 등

#### 01 코딩 테스트 효율적으로 준비하기
##### 언어 선택하기
- 언어는 사실 크게 중요한 요소는 아니다.
##### 문제 분석 연습하기
- 코딩 테스트는 코딩 능력이 아니라 문제 풀이 능력을 확인하는 것이다.
- 그래서 무작정 코드를 작성하기보다 문제 분석에 시간을 충분히 사용해야 한다.
- 2시간 ~ 4시간 정도 문제 풀 시간을 주므로 전체 시간의 50% ~ 60% 정도는 문제 분석에 시간을 쓰는 것이 좋다
###### 문제를 쪼개서 분석하라
- 문제 전체를 한번에 분석하는 것보다 동작 단위로 쪼개서 분석하는 것이 유리하다.
###### 제약 사항을 파악하고 테스트 케이스를 추가하라
- 제약 사항을 정리하고 고려하여 테스트 케이스를 추가하는 연습을 하는 것이 좋다
- 어떤 알고리즘을 사용할지 고민할 때 유용하고, 추후 코드를 구현하는 단계에서 예외를 거를 때 도움이 된다.

###### 입력값을 분석하라
- 보통 알고리즘의 시간 복잡도는 입력값이 결정하는 경우가 많다.
- 입력값의 크기를 확인하면 문제를 제한시간 내에 풀 수 있는 알고리즘과 그렇지 않은 알고리즘을 걸러낼 수 있다. 

###### 그리디하게 접근할 때는 근거를 명확히 해라
- 그리디 : 현재 상황에서 가장 유리해 보이는 선택을 하는 것
- 그리디는 반드시 수학적, 논리적으로 정답이 나올 수 있을지 검토한 후에 사용해야 한다.

###### 데이터 흐름이나 구성을 파악하라
- 데이터의 삽입과 삭제가 빈번하게 일어나는 상황에서 최솟값 최댓값을 반복해서 구해야 한다면 힙 자료구조를 고려한다.
- 그러나 데이터가 50개 미만이고 입력값을 깔끔하게 정리하기 어렵다면 하드 코딩을 고려한다.
- 데이터 값의 차이가 크면 데이터 값 자체를 배열의 인덱스로 활용하는 것은 피하는 것이 좋다.(메모리 낭비)

##### 의사 코드로 설계하는 연습하기
- 문제 분석이 끝난 다음에는 분석 내용을 바탕으로 전체적인 코드를 설계해야 한다(구현 X) => 의사 코드 작성
	- 의사 코드 : 프로그램의 논리를 설명하고 알고리즘을 표현하기 위해 작성한 일종의 지침
		1. 프로그래밍 언어로 작성X
		2. 일반인도 이해할 수 있는 자연어
		3. 일정 형식이 없음
- 추상 단계에서 설계를 진행할 수 있으므로 설계 아이디어에 집중할 수 있다.
- 의사 코드를 수정하는 것은 수정 시간이 더 짧아 시간에서도 이득을 볼 수 있다. 

###### 세부 구현이 아닌 동작 중심으로 작성하라
- 의사 코드는 동작 중심으로 작성해야 한다.
- 실제 프로그래밍 요소는 의사 코드에 추가하면 안된다.

###### 문제 해결 순서로 작성하라
- 의사 코드를 토대로 구현할 것이므로 문제 해결 순서대로 작성해야 한다.

###### 충분히 테스트 하라
- 구현 전 마지막 단계는 충분히 테스트 하라는 것이다.
- 구현 단계로 갈수록 잘못된 부분을 수정하는 데 드는 비용이 점점 커진다. 
- 의사 코드가 미리 생각해본 테스트 케이스를 통과할 수 있을지를 고민해봐야 한다. 
- 의사 코드가 대부분의 테스트 케이스를 통과할 수 있을 것 같을 때 구현을 시작한다.

#### 02 프로그래머스 완벽 활용 가이드
##### 프로그래머스는 어떤 곳인가요? 
- 코딩 테스트를 대비하기 위한 학습 코스 제공
- 여러 IT 기업들이 프로그래머스에서 코딩 테스트를 진행하므로 학습 사이트와 응시 사이트가 같을 확률이 높음
##### 프로그래머스 활용 가이드
###### 프로그래머스 살펴보기
- 코딩테스트 -> 스킬 체크
- 레벨 당 정확성/효율성 테스트 결과와 코딩 테스트 점수 항목을 볼 수 있음
- 코딩 테스트 > 코딩 테스트 문제에서는 난이도와 언어를 선택하여 문제를 선택할 수 있음
- 다른 사람 풀이를 볼 수 있음(단, 문제 풀지 않은 상태에서 보면 기본 점수 1000점에서 점수 차감)
- 제출 후 채점하기 이후 다른 사람 풀이를 편하게 볼 수 있음
- 채점 기준
	- 정확성 테스트: 제출한 코드 정답을 제대로 출력하는지 확인하는데, 정답이 맞으면서도 각 테스트 케이스 수행 시간이 10초 이내여야 함
	- 효율성 테스트: 알고리즘의 성능을 확인. 시간 복잡도(빅오)로 판단
#### 03 알고리즘 효율 분석
##### 시간 복잡도란?
- 문제들은 저마다 효율적으로 해결하는 알고리즘이 있다. 이는 알고리즘이 실행되는 제한 시간과 관련이 있다.
- 알고리즘 선정 기준 > 시간 복잡도
- 시간 복잡도(time complexity)
	- 알고리즘 성능을 나타내는 지표
	- 입력 크기에 대한 연산 횟수의 상한
	- 낮으면 낮을 수록 좋은 것
- 입력 크기 : 알고리즘이 처리해야 할 데이터 양(처리해야 할 데이터 개수. 숫자 크기 아님)

###### 알고리즘 수행 시간을 측정하는 방법
- 알고리즘 수행 시간 측정 방법으로 절대 시간을 측정하는 방법과 시간 복잡도를 측정하는 방법이 있다.
- 절대 시간을 측정: 말 그대로 시간을 측정하면 된다. 단, 프로그램을 실행하는 환경에 따라 달라질 수 있어서 코딩 테스트에서 잘 활용하지 않는다.
- 시간 복잡도를 측정
	- 시간 복잡도는 알고리즘이 시작한 순간부터 결괏값이 나올 때까지의 연산 횟수를 나타낸다. 
	- 최선, 보통, 최악의 경우로 나뉜다.
	- 입력의 크기를 N으로 일반화하여 연산 횟수의 추이를 나타내야 한다.
		- 이런 방식으로 입력 크기에 따른 연산 횟수의 추이를 활용해서 시간 복잡도를 표현하는 방법을 점근적 표기법이라고 한다.
	- 코딩 테스트에서 모든 경우의 수에서 알고리즘이 문제를 처리하는 것을 고려해야 하므로 시간 복잡도는 최악의 경우를 가정하여 이야기하는 것이 일반적이다. 

###### 최악의 경우 시간 복잡도를 표현하는 빅오 표기법
- 빅오 표기법(big-O notation)
	- 가장 많이 사용하는 점근적 표기법으로 상한선을 활용하는 방법이다. 
		- 점근 표기법 : 어떤 함수의 증가하는 추세를 표현하는 표기법
	- 어떤 프로그램의 연산 횟수가 f(x) 라고 할 때 함수의 최고차항을 남기고 계수를 지워 O(...)이라고 표기하면 된다. 
- 위와 같이 표기하는 이유
	- 이터가 커질 수록 차수의 격차는 심하게 벌어지므로 계수의 차이나 다른 차항은 무시할 수 있을 정도로 작다고 판단하는 것
- 로그함수는 다항함수보다 느리게 증가하고 지수함수는 다항함수보다 빠르게 증가한다. 따라서 최고차항만 남기는 작업에서 우선순위는 다음과 같다.
	- 지수함수: `2^x` 
	- 다항함수: `3x^2` 
	- 로그함수: logx
- 즉, 이 우선 순위대로 표기할 때 지수함수와 다항함수가 섞여있다면 다항함수를 지워야 한다.
- 빅오는 시간 복잡도의 점근적 상한으로 빅오 상한의 정의를 만족하는 시간 복잡도 함수들의 집합으로 해석할 수 있다.

###### 시간 복잡도를 코딩 테스트에 활용하는 방법
- 코딩 테스트 문제에는 제한 시간이 있으므로 문제를 분석한 후 빅오 표기법을 활용해서 해당 알고리즘을 적용했을 때 제한 시간 내에 출력값이 나올 수 있을지 확인해볼 수 있다.
- 연산 횟수는 1000 ~ 3000만 정도로 고려해서 시간 복잡도를 생각하면 된다.
- 아까와 같이 배열에서 검색하기로 돌아가면, 하나하나 짚어가며 찾는 방식은 시간 복잡도가 O(N)이다. 즉 데이터 갯수가 많지 않다면 해당 알고리즘은 사용해도 된다.

###### 박테리아 수명 문제
- 초기 박테리아 세포 개수가 N일 때 해마다 세포 개수가 이전 세포 개수의 반으로 준다면 언제 모든 박테리아가 죽을지 계산해야 한다. 예시로 N=16이면, 출력은 5이다.
- 푸는 과정
	- N이 16인 경우 모든 박테리아는 5년이면 소멸한다. 
	- 현재 박테리아의 수가 N이라면 1년 뒤의 박테리아의 수는 1/2 * N 이라고 할 수 있다.
	- Y년 후의 박테리아 수는 (1/2)^Y * N 이다.
	- 그러면 박테리아 소멸 시기는 (1/2)^Y * N 의 값이 최초로 1보다 작아질 때이므로 (1/2)^Y * N <= 1 인 Y를 찾으면 된다.

#### 04 코딩 테스트 필수 문법
- 대부분 내용 생략 
##### 코딩 테스트 코드 구현 노하우
###### 조기 반환(early return)
- 코드 실행 과정이 함수 끝까지 도달하기 전에 반환하는 기법
- 코드의 가독성을 높여주고 예외를 깔끔하고 빠르게 처리

###### 보호 구문(guard clauses)
- 본격적인 로직을 진행하기 전 예외 처리 코드를 추가하는 기법
- 예를 들어 조건문을 이용하여 초기에 입력값이 유효한지 검사하고 그렇지 않으면 바로 함수를 종료

###### 제네릭(generic)
- 빌드 레벨에서 타입을 체크하여 타입 안정성을 제공하고 타입 체크와 형변환을 생략할 수 있게 해줘 코드를 간결하게 만들어주는 기능
- 코딩 테스트에서는 여러 타입의 데이터를 하나의 컬렉션에 넣어야 하는 경우는 거의 없으므로 제네릭으로 타입을 강제하여 실수를 방지하는 것이 좋다

###### 스트림
- 데이터의 흐름으로 데이터 소스를 추상화 하고 데이터를 다루는 데 유용한 메서드를 정의해놓은 것이다.
- 코드의 양을 줄이고 가독성을 향상 시킬 수 있다.
- 단일 스레드 환경의 코딩 테스트 관점에서 시간 복잡도 측면에서 for문등의 반복문과 스트림은 성능에 큰 차이가 없으므로 편한 것으로 쓰면 된다.
